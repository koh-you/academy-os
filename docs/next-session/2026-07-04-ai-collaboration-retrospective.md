# 2026-07-04 AI Collaboration Retrospective

이 문서는 다음 Codex 세션이 오늘의 작업 방식과 판단 기준을 이어받기 위한 별도 회고입니다. 핵심은 시험분석을 오래된 구조 위에 계속 보정하지 않고, 테스트 단계라는 점을 인정한 뒤 삭제하고 v2로 다시 만든 것이 더 빠르고 안정적이었다는 점입니다.

## 오늘 작업 요약

1. 기존 시험분석 구조를 되살리지 않고 v2 파이프라인으로 재구성했다.
   - 레거시 시험분석 탭, 레거시 AI/PDF API, `app_state.examAnalyses` 중심 흐름은 복구하지 않는 기준을 유지했다.
   - 새 흐름은 `PDF 저장 -> 텍스트/페이지 추출 -> Claude 원본 검증 -> 문항 수 판독 -> 선생님 확인 -> 1~N 행 고정 -> 문항 경계 -> AI 행 채움 -> AI 결과 검수 -> 최종 미리보기`다.

2. 테스트 PDF 기준으로 작은 단계별 검증을 통과했다.
   - PDF 업로드와 Storage 저장을 확인했다.
   - 텍스트 추출은 5쪽, 약 52KB, 문항 후보 1~24를 잡았다.
   - Claude 원본 검증은 5쪽, 공통수학1, 24문항 후보로 성공했다.
   - 선생님이 24문항을 확정하고 1~24행을 생성했다.
   - 문항 경계 탐지, AI 행 채움, AI 2차 수정, 새로고침 후 저장 유지까지 확인했다.

3. `AI 결과 검수`를 실제 선생님 수정 화면으로 정리했다.
   - 카드형이 아니라 엑셀식 한 행 구조로 바꿨다.
   - 단원, 쎈 주유형, 보조유형, 난이도, 검수 메모, 확정 체크를 이 표에서 수정한다.
   - 수정 후 `시험분석 · 수정됨 · 저장 필요`, 저장 중, 저장 완료 상태가 같은 패널 안에 보인다.

4. 최종 미리보기는 읽기 전용으로 정리했다.
   - 최종 미리보기는 도넛 차트, 막대 차트, 문항 흐름, 주요문항 후보, 상세 표를 보여준다.
   - 난이도 수정 select와 저장 버튼은 제거했다.
   - 난이도 수정과 저장은 위 `AI 결과 검수` 표에서만 한다.
   - `0/24` 같은 확정 카운트는 저장 여부로 오해되므로 최종 미리보기 헤더에서는 숨겼다.

5. 큰 개발 방향을 작업지침에 고정했다.
   - 기본 구조는 `AI 초안 -> 사람이 보기 쉬운 편집 화면 -> 사람 수정/검수 -> 검수본 원본화 -> 산출물/누적데이터 반영`이다.
   - AI/템플릿/자동 매핑은 최초 초안일 뿐이다.
   - 사용자가 수정한 뒤에는 그 수정본이 새 원본이다.
   - 새로고침, 자동매핑, 렌더 재계산이 사용자 편집본을 다시 덮어쓰면 안 된다.

6. 운영 반영 지연도 확인했다.
   - 로컬 코드와 로컬 dev 서버에서는 미리보기의 `난이도 수정 저장` 버튼이 제거되어 있었다.
   - 운영 Vercel은 잠시 직전 번들을 내려줘서 사용자가 버튼을 계속 볼 수 있었다.
   - 새 커밋으로 배포를 다시 트리거했고, 운영 번들에서 버튼 제거와 새 팔레트 반영을 확인했다.

## 오늘의 중요한 판단

### 오래 안 되는 테스트 기능은 삭제 후 재구성이 더 나을 수 있다

이번 시험분석은 3일 동안 레거시 구조 위에서 보정, fallback, mock 의심, 저장 원천 혼동, PDF 추출 한계, AI 응답 병합 문제가 계속 겹쳤다. 이때 문제를 숨기는 보정 레이어를 더 쌓으면 다음 오류가 더 찾기 어려워진다.

테스트 단계라면 다음 기준에서 삭제 후 재구성이 낫다.

- 기존 데이터가 운영 원본이 아니라 테스트 데이터다.
- 저장 원천이 여러 개라 어느 값이 진짜인지 계속 헷갈린다.
- 화면 표시, 저장 row, AI 응답, mock/fallback, local draft가 서로 다른 말을 한다.
- 필터, normalize, 우선순위 보정이 2겹 이상 필요해진다.
- 사용자가 원하는 실제 workflow가 기존 구조와 다르다.
- 기능 크기가 너무 크지 않아 다시 만드는 편이 빠르다.

이번에는 레거시 시험분석을 지우고 v2를 작게 다시 만들면서, 오히려 다음 단계가 선명해졌다. AI 입장에서도 삭제 후 재구성은 기억해야 할 선택지다. 단, 운영 데이터가 섞여 있거나 되돌릴 수 없는 삭제라면 반드시 사용자 확인과 백업 기준이 먼저다.

## 병목과 해결 방식

### 1. PDF를 대화세션처럼 읽지 못하는 문제

병목:
- 웹앱 API 호출은 ChatGPT 대화세션의 PDF helper와 같지 않았다.
- 단순 텍스트 추출은 수식, 그래프, 인코딩 잡음에 취약했다.
- 문항별 crop/vision은 비용이 크고 효율이 낮았다.

해결:
- 텍스트 추출은 원문 분석 엔진이 아니라 품질 지표로 격하했다.
- Claude PDF `document` 입력을 얕은 원본 검증에만 사용했다.
- 지금 단계의 AI 검증 목적을 `쪽수`, `과목`, `문항 수 후보`, `누락 번호`, `정답표 감지`로 제한했다.
- 문항별 상세 분류는 문항 수와 1~N 행이 고정된 뒤에 진행했다.

효과:
- PDF 문제를 한 번에 해결하려 하지 않고, 먼저 원본이 읽히는지와 문항 수가 맞는지를 검증할 수 있었다.

### 2. 문항 수와 행 생성이 섞이는 문제

병목:
- 기존 구조에서는 저장값, AI 응답 길이, 텍스트 추론값, mock 의심값이 섞였다.
- AI가 문항 수와 문항별 내용을 한 번에 추론하면, 틀린 개수가 이후 모든 행을 오염시킨다.

해결:
- `문항 수 판독 -> 선생님 확인 -> 1~N 행 고정`을 독립 단계로 만들었다.
- 선생님이 24를 확정한 뒤에 1~24행을 생성했다.
- 이후 AI는 행 개수를 결정하지 않고, 고정된 행을 채우기만 한다.

효과:
- 새로고침 후에도 24행 기준이 유지됐다.
- AI가 21개만 채워도 24행 구조가 무너지지 않았다.

### 3. AI 초안이 사용자 수정본을 덮는 문제

병목:
- 자동초안/자동매핑 값이 textarea/input의 value를 매 렌더마다 다시 계산하면 커서, 엔터, 사용자 수정이 깨진다.
- AI 초안과 사용자 편집본의 우선순위가 흐려지면 새로고침 후 다시 AI 값으로 돌아갈 수 있다.

해결:
- `ai_fields -> localDraft -> teacher_fields/human_fields -> final_fields` 레이어를 명시했다.
- 사용자가 한 번 수정하면 그 수정본이 다음 원본이 되도록 작업지침에 고정했다.
- 최종 미리보기는 AI가 숫자를 다시 만들지 않고 저장된 검수본만 읽게 했다.

효과:
- 선생님 수정값이 새로고침 후 유지됐다.
- 미리보기와 편집 화면의 책임이 분리됐다.

### 4. 저장 중/저장 완료가 보이지 않는 문제

병목:
- 저장 버튼이 없거나 상태가 우상단에만 있으면 실제 저장 여부를 알기 어렵다.
- 차트가 즉시 바뀌면 저장된 것처럼 착각할 수 있다.

해결:
- 편집 패널 가까이에 `수정됨`, `저장 중`, `저장 완료`, `저장 실패`를 표시하는 원칙을 세웠다.
- 최종 미리보기에서 편집 기능을 제거하고, 저장 상태는 `AI 결과 검수` 패널에서만 보이게 했다.

효과:
- 사용자가 어디에서 수정하고 저장해야 하는지 명확해졌다.

### 5. 운영 반영이 늦게 보이는 문제

병목:
- 코드와 로컬 dev 서버는 맞는데 운영 화면은 이전 번들을 보여줄 수 있다.
- 이 경우 사용자는 "아직 버튼이 보인다"고 느낀다.

해결:
- 운영 URL의 실제 JS asset을 열어 문자열을 확인했다.
- 새 번들에서 `난이도 수정 저장`이 사라졌는지, 새 팔레트 문자열이 있는지 확인했다.

효과:
- 코드 문제와 배포/캐시 문제를 구분할 수 있었다.

## AI 입장에서 효율이 좋았던 사용자 프롬프트

### 1. "필터를 여러 겹 덧대지 마세요"

효과가 컸다. 이 문장 때문에 AI가 화면 표시만 맞추는 fallback/normalize 추가를 멈추고, 원천 데이터와 저장 경계를 보게 됐다.

좋은 형태:

```text
테스트 단계이니 기존 데이터가 손상되면 삭제하고 처음부터 다시 하면 됩니다.
필터/정규화/우선순위 보정을 여러 겹 덧대지 말고, 원천이 틀리면 원천을 고치거나 삭제 후 재생성하는 방향으로 봐주세요.
```

### 2. "검증가능한 절차를 알려주세요"

효과가 컸다. AI가 기능을 만들고 끝내지 않고, 사용자가 실제 화면에서 어떤 버튼을 눌러 성공/실패를 판단할지 기준을 만들게 했다.

좋은 형태:

```text
이 작업이 끝나면 제가 화면에서 무엇을 눌러보고, 어떤 문구/저장값이 나오면 통과인지 순서대로 알려주세요.
```

### 3. "AI 스스로도 그 질문에 답할 수 있는지 확인해주세요"

효과가 컸다. 단계별 구현만 하지 않고 전체 맥락, 사용자 의도, 저장 원천, 중단 조건을 함께 점검하게 했다.

좋은 형태:

```text
작업 후 AI 자기검토도 해주세요.
전체 맥락, 사용자 의도, 변경 이유, 저장 원천, 사용자 편집본 보호, 중단 조건에 답할 수 있어야 합니다.
```

### 4. "이 화면에서 제가 수정하는 곳인데 저장되는지 모르겠습니다"

효과가 컸다. UI 문제를 "버튼 추가"가 아니라 "저장 피드백과 작업 영역의 거리" 문제로 볼 수 있었다.

좋은 형태:

```text
이 화면은 제가 데이터를 수정하는 곳입니다.
저장 중/저장 완료가 같은 작업 영역 안에 보여야 합니다.
저장 버튼이 없거나 자동저장이라면 상태 표시를 분명히 해주세요.
```

### 5. "코딩 전에 이 질문들에 답해주세요"

효과가 컸다. 산출물, 차트, 블로그/인스타/PPT/Canva 같은 큰 방향은 바로 구현하면 흔들릴 수 있었는데, 먼저 결정할 문제를 분리할 수 있었다.

좋은 형태:

```text
코딩 전에 이 3가지 질문에 답해주세요.
1. 앱 안 미리보기는 어떻게 렌더링할지
2. 내부 누적데이터는 나중에 어떻게 연결될지
3. 블로그/인스타 산출물 방향은 무엇인지
```

### 6. 스크린샷과 짧은 요구를 같이 주는 방식

효과가 컸다. "우상단이 잘 안 보입니다", "0/24가 뭔가요", "난이도수정버튼이 여전히 있습니다"처럼 화면 기준으로 말하면 AI가 코드와 실제 배포 상태를 대조할 수 있었다.

좋은 형태:

```text
스크린샷 기준으로 여기 버튼이 아직 보입니다.
이 버튼은 없어야 하고, 수정은 위 표에서만 하겠습니다.
코드뿐 아니라 운영 화면에 실제 반영됐는지도 확인해주세요.
```

## 다음 작업 후보

### 1순위: 주요문항 추천/선생님 선택 저장

목표:
- AI가 특이 문항, 변별 문항, 설명 가치가 큰 문항 후보를 추천한다.
- 선생님이 후보를 선택/해제하고 저장한다.
- 최종 미리보기, 블로그, 인스타 산출물은 선생님 선택값을 원본으로 사용한다.

주의:
- `주유형 TOP`이 아니라 `주요문항`이다.
- AI 추천은 초안이고, 선생님 선택값이 원본이다.
- 추천/선택 저장 상태가 같은 패널 안에 보여야 한다.

검토 gate:
- AI 추천 후보가 뜬다.
- 선생님이 체크/해제할 수 있다.
- 저장 후 새로고침해도 선택값이 유지된다.
- 최종 미리보기의 주요문항 후보 영역이 저장된 선택값을 우선한다.

### 2순위: 블로그 산출물 데이터 모델

목표:
- 네이버 블로그형 구조를 통일 템플릿으로 만든다.
- 시험 기본정보, 한 줄 총평, 단원별 특징, 주요문항 포인트, 대비 방법, 이미지 슬롯을 포함한다.

주의:
- 공개 산출물은 이미지 중심이다.
- 원문 이미지 슬롯은 사용자가 직접 crop하거나 나중에 HWP/HWPX/PDF/Canva 경로를 검증한다.
- 앱 안 분석 시각화와 공개용 블로그/인스타 이미지는 별도다.

### 3순위: 인스타/Canva/PPT 카드뉴스 구조

목표:
- 색상코드, 카드 구성, 이미지 슬롯을 고정한다.
- 이후에는 내용만 바꿔 반복 제작할 수 있게 한다.

주의:
- Canva 연결은 나중에 템플릿이 확정된 뒤 진행한다.
- 먼저 앱 안에서 어떤 데이터가 카드로 갈지 정한다.

### 4순위: 데이터 원천 중복/덧대기 전수조사

목표:
- fallback, mock, sample, legacy, app_state, localStorage, normalize, dedupe가 운영 원천을 흐리는 곳이 있는지 조사한다.

주의:
- 바로 고치기보다 후보 표를 만든 뒤 사용자와 우선순위를 확정한다.
