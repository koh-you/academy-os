# Academy OS UI UX 안정화 작업 계획 - 2026-06-21

이 문서는 Academy OS가 현재 기능 확장 단계에서 UI/UX 안정화와 운영 흐름 다듬기 단계로 넘어가면서, 추가로 정리해야 할 작업들을 전체 맥락에서 정리한 문서입니다.

## 1. 현재 판단

현재 앱은 기능이 부족한 단계라기보다, 기능들이 서로 연결되면서 사용자에게 보이는 의미가 흔들리는 단계입니다.

특히 아래 영역에서 혼란이 반복되었습니다.

- 같은 상태가 화면마다 다른 색상으로 보임
- 같은 버튼처럼 보이지만 실제 의미가 다름
- 저장됨, 작성됨, 예약됨, 발송됨, 테스트 기록됨의 구분이 모호함
- 실제 Supabase에 저장된 데이터와 화면용 파생 데이터가 섞여 보임
- 시험관리, 학사일정, 수업일지, 알림톡, 보충관리의 데이터 원천이 사용자가 보기에는 불명확함

따라서 다음 작업은 새 기능 추가보다 다음 원칙을 맞추는 것이 우선입니다.

- 같은 상태는 같은 색
- 같은 버튼은 같은 의미
- 같은 데이터는 같은 원천
- 저장 여부와 발송 여부는 명확히 분리
- 실제 저장 데이터와 화면용 가상 데이터는 구분

## 2. 상태 표시 체계 통일

가장 먼저 전체 앱의 상태 색상과 문구를 통일해야 합니다.

### 알림톡

- 미작성
- 작성됨
- 예약 중
- 테스트 발송 기록됨
- 발송 완료
- 발송 실패

학생 알림톡과 학부모 알림톡은 반드시 같은 색상 규칙을 따라야 합니다.

### 보충관리

- 진행 중
- 추가보충 필요
- 보충 완료
- 통과
- 복귀 가능

보충 통과와 삭제는 의미가 다르므로 색상과 문구를 분리해야 합니다.

### 시험관리

- 미입력
- 일부 입력
- 입력 완료
- 자동생성 후보
- 저장됨
- 자동생성 제외

시험관리 데이터는 학사일정과 자동수업 생성의 원천이므로, 입력 상태가 눈에 잘 보여야 합니다.

### 수업일지

- 작성 전
- 저장됨
- 알림톡 반영됨
- 직전 수업메모 있음

수업메모가 저장되었는지, 알림톡 초안에 반영되었는지는 분리해서 보여야 합니다.

## 3. 버튼 역할 정리

버튼이 많아지면서 역할이 겹치는 부분이 생겼습니다. 버튼명과 동작을 다음 기준으로 정리하는 것이 좋습니다.

| 버튼명 | 의미 |
| --- | --- |
| 저장 | 현재 모달 내용만 저장 |
| 저장 및 반영 | 저장 후 관련 화면, 알림톡, 캘린더까지 반영 |
| 알림톡 작성 | 알림톡 문구 작성 화면 열기 |
| 예약 | 실제 발송 예약 |
| 테스트 발송 | 실제 발송 없이 테스트 기록 |
| 삭제 | 실제 삭제 또는 행 제거 |
| 취소 처리 | 화면에서는 숨기고 DB에는 일정 기간 보관 |
| 자동생성 제외 | 자동 후보에서만 제외 |

특히 보충관리의 `알림톡 문구 작성`과 `수업일지 수정 반영`처럼 실무상 같은 흐름으로 움직이는 버튼은 하나로 합치는 방향이 맞습니다.

추천 문구는 `저장 및 알림톡 반영` 또는 `저장하고 일정 반영`입니다.

## 4. 데이터 원천 정리

오늘 오류가 많았던 이유 중 하나는 사용자가 보는 화면에서 데이터 원천이 섞여 보였기 때문입니다.

앞으로 각 데이터의 원본을 다음처럼 고정하는 것이 좋습니다.

| 데이터 | 원천 |
| --- | --- |
| 시험기간 | 시험관리 |
| 수학시험일 | 시험관리 |
| 학사일정 시험 표시 | 시험관리에서 파생 |
| 일요시험보강 실제 수업 | lessons |
| 일요시험보강 학교별 블록 수정 | app_state.generatedLessonControls.sundayMakeupBlocks |
| 이동된 학교별 블록 캘린더 표시 | 화면용 가상 lesson |
| 학생별 수업메모 | lesson student record |
| 보충 상태와 보충 메모 | makeup_tasks |
| 알림톡 문구와 발송 상태 | notification 관련 저장 데이터 |

화면에서는 이 원천을 전부 노출할 필요는 없지만, 상세 모달 안에서는 작은 보조문구로 `자동 생성 수업`, `학교별 블록`, `실제 저장 수업` 정도는 구분해주는 것이 좋습니다.

## 5. 모달 레이아웃 다듬기

가장 많이 쓰는 모달부터 안정화해야 합니다.

우선순위는 다음과 같습니다.

1. 수업메모 모달
2. 보충관리 상세 모달
3. 알림톡 작성 모달
4. 일요시험보강 블록 수정 모달
5. 시험관리 행 수정 UI

각 모달에서 확인할 기준은 같습니다.

- 저장 버튼 위치가 항상 예측 가능한가
- 저장됐는지 즉시 보이는가
- 긴 텍스트가 잘리지 않는가
- 버튼 색상과 크기가 통일되어 있는가
- 작성, 저장, 발송, 반영의 의미가 분리되어 있는가
- 모바일 또는 좁은 화면에서도 핵심 입력칸이 너무 작아지지 않는가

## 6. 목록과 표 간격 재조정

시험관리처럼 표가 너무 넓게 벌어지는 문제는 다른 운영 화면에서도 반복될 수 있습니다.

추가로 점검할 화면은 다음과 같습니다.

- 시험정보 탭
- 보충관리 목록
- 최근 한 달 보충내역
- 수업일지 학생별 상세
- 알림톡 상태 열
- 학사일정 캘린더 pill 표시

학원 운영툴은 마케팅 페이지처럼 넓은 여백보다, 한눈에 비교 가능한 촘촘하고 정돈된 화면이 더 적합합니다.

## 7. 삭제와 복구 UX 정리

현재 삭제처럼 보이는 동작이 여러 종류입니다.

- 시험관리 행 삭제
- 일요보강 블록 삭제
- 수업 취소 후 7일 보관
- 보충 통과 후 복귀
- 자동생성 제외
- 중복 데이터 삭제

이 동작들은 같은 `삭제` 버튼으로 보이면 안 됩니다.

추천 기준은 다음과 같습니다.

| 동작 | 권장 문구 |
| --- | --- |
| DB 행 제거 | 삭제 |
| 일정 기간 보관 후 숨김 | 취소 처리 |
| 자동 후보만 숨김 | 자동생성 제외 |
| 완료 상태 되돌리기 | 보충관리로 복귀 |
| 중복 데이터 제거 | 중복 삭제 |

삭제 전 확인 문구에는 실제 DB에서 삭제되는지, 화면에서만 숨겨지는지, 되돌릴 수 있는지를 반드시 적어야 합니다.

## 8. 알림톡 흐름 QA

알림톡은 실제 발송 리스크가 있으므로 별도 QA 체크리스트로 관리해야 합니다.

확인할 항목은 다음과 같습니다.

- 학생용과 학부모용이 같은 상태 색상 규칙을 따르는가
- 테스트 발송과 실제 예약이 명확히 구분되는가
- 예약만 했는데 테스트 발송 기록으로 보이지 않는가
- 문구를 수정하면 발송완료 상태가 작성됨 상태로 돌아가는가
- 발송 실패 시 빨간색으로 표시되는가
- 수업메모와 보충메모가 중복 삽입되지 않는가
- 긴 문구에서도 미리보기와 실제 발송 문구가 크게 다르지 않은가

## 9. 운영용 확인 표시

개발자용 화면까지는 아니어도, 운영 중 확인이 어려운 데이터는 최소한 상세 화면에서 볼 수 있어야 합니다.

후보는 다음과 같습니다.

- 이 알림톡이 작성됨, 예약됨, 테스트 기록됨, 실제 발송 완료 중 무엇인지
- 이 일요보강이 실제 lessons인지, 가상 파생 블록인지
- 이 보충수업이 makeup_tasks만 있는지, lessons까지 생성됐는지
- 이 수업메모가 Supabase 저장 완료인지

이 정보는 목록에 과하게 노출하지 말고, 상세 모달의 보조문구나 상태 배지로만 보여주는 것이 적합합니다.

## 10. 추천 작업 순서

전체 UI/UX 안정화 작업은 다음 순서로 진행하는 것이 좋습니다.

1. 상태 색상과 문구 통일
2. 버튼 역할 통일
3. 수업메모, 알림톡, 보충관리 데이터 흐름 안정화
4. 시험관리와 일요보강의 실제 수업/가상 블록 구분 정리
5. 표, 모달, 버튼 크기와 간격 통일
6. 삭제/복구 UX 문구 정리
7. 운영 체크리스트 기반 회귀 테스트 추가

## 11. 다음 작업 큐 후보

### P0. 상태 색상 시스템 정리

- 알림톡 상태 색상 기준을 상수화한다.
- 학부모/학생 알림톡 버튼이 같은 함수와 같은 CSS class를 쓰게 한다.
- 보충관리와 수업일지의 저장/완료/실패 색상도 같은 계열로 맞춘다.

### P1. 버튼 컴포넌트 정리

- primary, secondary, success, warning, danger, ghost 버튼 스타일을 정리한다.
- 같은 의미의 버튼은 같은 크기와 색을 쓰게 한다.
- 모달 하단 버튼 위치를 통일한다.

### P1. 보충관리 통합 저장 QA

- 보충할 숙제 내역, 보충 메모, 보충 방식, 배정일을 수정한다.
- 저장 시 수업일지와 알림톡 초안에 동시에 반영되는지 확인한다.
- 보충 통과와 복귀 흐름을 다시 확인한다.

### P1. 수업메모 알림톡 반영 QA

- 6월 19일 수업메모가 6월 22일 직전 수업메모로 보이는지 확인한다.
- 수업메모가 알림톡 초안에 한 번만 들어가는지 확인한다.
- 문구 수정 후 상태가 작성됨으로 돌아가는지 확인한다.

### P2. 시험관리/일요보강 표시 안정화

- 학교별 블록이 다른 날짜로 이동하면 이동한 날짜에는 해당 학교만 보이게 한다.
- 원래 날짜에는 빠져나간 학교를 제외한 나머지만 보이게 한다.
- 블록 삭제 후 새로고침해도 유지되는지 확인한다.
- 실제 Supabase lessons에 중복 저장되지 않는지 확인한다.

### P2. 운영 체크리스트 문서화

- 수업일지 저장 확인
- 알림톡 예약/테스트/실패 확인
- 보충관리 통과/복귀 확인
- 시험관리 행 삭제/일요보강 연동 확인
- SQL Editor 필요 여부 확인

## 12. 배포와 SQL Editor 유의점

- 코드 수정 후에는 `npm run build`를 실행한다.
- 운영 흐름에 영향을 주면 `npm run test:production`도 실행한다.
- 변경 내용은 `docs/current-worklog.md`에 남긴다.
- GitHub `main`에 push되어야 Vercel/Render 자동 배포 흐름에 올라간다.
- Supabase SQL migration은 자동 적용된다고 가정하지 않는다.
- DB 컬럼, 테이블, RPC, 정책 변경이 필요한 경우 Supabase SQL Editor에서 사용자가 직접 실행해야 한다.
- SQL이 필요 없을 때도 최종 보고에 `SQL Editor 작업 필요 없음`이라고 명확히 적는다.
- 비밀키, `.env`, Supabase service role key, Render/Vercel/GitHub 토큰은 절대 출력하거나 commit하지 않는다.
