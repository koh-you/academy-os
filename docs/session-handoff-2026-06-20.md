## 2. SQL Editor 관련 정리

오늘 사용자가 Supabase SQL Editor에 수동 적용한 SQL이 있었고, 이후 확인 요청이 있었다.

확인된 원칙:

- Supabase는 이 웹앱의 운영 데이터베이스다.
- GitHub `main` push는 Vercel/Render 자동 배포로 이어지지만, Supabase SQL migration은 repo 기준 자동 적용이 확인되지 않았다.
- 따라서 DB 컬럼/테이블 변경이 필요한 작업은 사용자가 Supabase SQL Editor에서 직접 적용해야 한다.
- SQL 작업이 필요할 때는 반드시 사용자에게 별도로 안내하고, `docs/current-worklog.md`에도 남긴다.

오늘 후반 작업 중 SQL Editor가 필요 없었던 작업:

- 보충관리 UI/메모/알림톡 초안 반영
- 일요시험보강 학교별 블록 수정
- 이동된 일요시험보강 블록을 달력에 파생 표시
- 시나리오 테스트 61~64 추가

이유:

- 기존 `makeup_tasks.note` JSON 메타데이터 또는 기존 `app_state` JSON을 활용했다.
- 새 Supabase 컬럼/테이블을 요구하지 않았다.

다음 세션 체크:

- 새 기능이 DB 컬럼을 요구하면 SQL 파일 여부를 먼저 확인한다.
- SQL 필요 시 적용 파일명과 순서를 사용자에게 초보자 기준으로 안내한다.
- 비밀키, `.env`, Supabase service role key는 절대 출력하거나 commit하지 않는다.

## 3. 수업일지 - 보충관리 / 결석일정

가장 우려했던 문제:

- 숙제 검사 결과가 완료인데도 보충관리 목록에 올라오는 문제
- 미완료/50%/80%/미검사만 보충관리 대상이어야 하는 문제
- 보충 통과를 실수로 누르면 최근 한 달 보충내역에서 되돌릴 수 있어야 하는 문제
- 보충 일정과 알림톡 문구, 수업일지 상세가 서로 다른 데이터를 보는 문제
- 보충을 했지만 다 끝나지 않은 경우 진행 메모와 추가보충 추적이 필요한 문제

오늘 처리된 내용:

- 숙제보충 후보는 수업일지에서 체크된 지난 숙제 결과를 기준으로 필터링한다.
- 숙제보충 내역은 보충관리 상세 모달에서 수정 가능하다.
- 수정한 숙제보충 내역은 알림톡 초안 재료로 사용한다.
- 수정한 숙제보충 내역은 수업일지의 숙제보충 상세 화면에도 반영된다.
- 보충 통과 처리에는 확인 모달을 추가했다.
- 최근 한 달 보충내역에서 실수로 통과 처리한 항목을 다시 보충관리로 되돌릴 수 있다.
- 보충 메모는 `supplementProgressMemo` 성격으로 저장/표시한다.
- 보충 처리 상태는 진행 중, 추가보충 필요, 보충 완료 흐름으로 정리했다.
- 추가보충이 필요하면 완료 처리하지 않고 계속 추적할 수 있게 했다.
- 보충 방식 순서:
  - 숙제보충: `남아서 하고가기 -> 다음시간까지 -> 등원보충`
  - 결석보강: `녹강보강 -> 현장보강`

다음 세션 확인 체크리스트:

- 6월 19일 숙제 미완료 학생이 보충관리 목록에 뜨는지 확인
- 완료 숙제 학생은 보충관리 목록에 뜨지 않는지 확인
- 보충관리에서 숙제보충 내역 수정 후 알림톡 초안에 반영되는지 확인
- 수업일지에서 해당 보충수업을 열었을 때 수정 내역이 보이는지 확인
- 보충 통과 처리 후 최근 한 달 내역에서 취소/복귀가 되는지 확인
- 보충 진행 메모가 목록과 상세에서 일관되게 보이는지 확인

## 4. 수업일지 - 시험관리 - 학사일정 캘린더 중복 문제

가장 우려했던 문제:

- 학사일정에서 내신기간과 수학시험날짜를 같이 저장하면 수동 이벤트와 시험관리 파생 이벤트가 동시에 보여 중복될 수 있었다.
- 시험기간/수학시험날짜가 학사일정과 시험관리 양쪽에서 수정되면 어느 쪽이 원본인지 흐려질 수 있었다.
- 자동생성 수업을 수동수정했을 때 다음 자동생성이 다시 덮어쓰거나 중복 생성할 수 있었다.
- 특정 학교의 시험기간이 갑자기 모든 학교 시험기간처럼 보이는 문제가 있었다.

현재 권장 운영 방식:

- 시험기간과 수학시험날짜는 `시험관리`에서 입력한다.
- 방학, 개학, 모의고사, 학교 행사, 단순 캘린더 이벤트는 `학사일정`에서 입력한다.
- 학사일정은 시험관리 데이터를 보여줄 수는 있지만, 시험 관련 데이터의 원천은 시험관리로 둔다.

오늘 처리된 구조:

- 시험관리/학사일정 기반 자동 수업 후보를 만든다.
- 자동 수업 후보는 곧바로 무조건 저장하지 않고, 후보 패널에서 확인 후 반영할 수 있다.
- 자동 수업에는 `generatedKey`를 둬서 같은 후보가 중복 생성되지 않게 한다.
- 사용자가 자동생성 수업을 직접 수정하면 `manualOverrideKeys`로 보호할 수 있다.
- 사용자가 자동생성 후보를 제외하면 `suppressedKeys`로 다시 생성되지 않게 할 수 있다.
- 일요시험보강의 학교별 세부 시간표는 원본 lesson을 여러 개로 쪼개 저장하지 않고, `generatedLessonControls.sundayMakeupBlocks`에 수동수정값을 분리 저장한다.
- 학교별 블록을 다른 날짜/시간으로 옮기면, 실제 `lessons` 중복 저장 없이 달력 표시용 가상 수업으로 파생한다.
- 이동된 블록을 클릭하면 원본 일요시험보강 모달이 열리고 해당 블록이 강조된다.

시나리오 테스트로 확인된 항목:

- 61 moved Sunday makeup blocks derive calendar lessons
- 62 moved Sunday makeup blocks keep original lesson linkage
- 63 moved Sunday makeup blocks open focused source modal
- 64 moved Sunday makeup blocks are visually distinct

다음 세션 확인 체크리스트:

- 시험관리에서 학교별 시험기간을 입력하면 학사일정에 시험기간이 보이는지 확인
- 시험관리에서 날짜별 수학시험을 입력하면 학사일정에 날짜별 수학시험이 보이는지 확인

## 5. 현재 남은 UI/UX 검토 포인트

치명적 오류는 현재 테스트에서 보이지 않는다. 다만 다음 개선 후보가 있다.

1. 이동된 일요시험보강 블록을 선택했을 때 키보드 단축키 `Del`, `Ctrl+C`의 의미가 애매하다.
   - 가상 수업은 실제 lesson이 아니므로 삭제/복사 단축키를 비활성화하거나 안내 문구를 바꾸는 것이 좋다.

2. 학교별 블록을 다른 날짜로 옮겨도 원래 날짜의 통합 일요시험보강 pill에는 전체 학교명이 남아 있을 수 있다.
   - 데이터 중복은 아니지만 화면상으로는 중복처럼 느껴질 수 있다.
   - 다음 개선: 원본 pill에는 그 날짜에 남아 있는 블록만 표시한다.

3. `자동 수업 후보` 패널은 기능적으로 필요하지만 화면이 복잡해질 수 있다.
   - 운영이 안정되면 기본 접힘 상태 또는 고급 설정 영역으로 보내는 것을 검토한다.
