# Academy OS Agent Guide

이 파일은 Codex 새 세션이 이 저장소에서 작업할 때 가장 먼저 읽는 프로젝트 지침입니다.

## Remaining Human Gates - Always Report Before Saying Work Is Done

상태 보고, 진행 보고, 작업 완료 답변에서는 완료한 항목만 나열하지 않는다. 답변 상단에 `남은 사람 gate`를 별도 제목으로 먼저 표시하고, 각 gate가 무엇을 막고 있는지까지 명시한다.

- 새 세션 시작 답변에서는 `미룬 작업 큐`를 먼저 보여준 직후 `남은 사람 gate`를 이어서 표시한다. 사용자가 `어디까지 진행됐나요`, `현재 상태`, `다음은 무엇인가요`처럼 상태를 물으면 답변 첫 부분에 남은 gate부터 표시한다.
- gate마다 `검토 대상`, `사용자가 실제로 확인할 순서`, `통과 기준`, `실패/중단 조건`, `통과 전 금지되는 다음 단계(commit/push/배포/다음 의미 단위)`를 짧고 구체적으로 쓴다.
- `착수 전 gate`, `코드 이동 후 gate`, `배포 후 운영 gate`, `별도 보안 gate`, `사용자가 보류한 gate`를 구분한다. 보류나 미실시는 통과로 표현하지 않는다.
- AI 자동검증이 모두 통과해도 사람 gate가 남아 있으면 `완료`라고만 말하지 않는다. `코드/자동검증 완료 · 사람 gate 대기 · commit/push 대기`처럼 현재 단계와 blocked next action을 함께 말한다.
- 남은 gate가 없다면 `남은 사람 gate: 없음`이라고 명시한다. 새 gate가 생기거나 통과하면 이 섹션의 현재 목록, `docs/current-worklog.md`, 필요 시 `docs/next-session/README.md`를 함께 갱신한다.

현재 리팩터링 세션의 남은 사람 gate:

1. `학생 포털 실제 쓰기 검수` — 숙제 완료, 질문 CRUD, 시험 후 제출의 저장·새로고침·재로그인·강사 미리보기 차단·교사 확인을 실제 학생으로 검수한다. 사용자 보류 상태이며 11B 리팩터링과는 별개다.
2. `교사 bearer session + Storage 소유권 보안 gate` — 교사 API 인증과 시험지 파일 열람 권한을 별도 고위험 작업으로 검증한다. 포털 표시 리팩터링이나 11B와 섞지 않으며 현재 통과가 아니다.
3. `Solapi 특강 템플릿 외부 검수` — 검수 완료 여부를 사용자에게 확인한다. 완료 전에는 템플릿 ID/변수 연결이나 테스트 발송을 진행하지 않으며, 이 리팩터링 세션에서는 완료 여부와 관계없이 연결 작업을 하지 않는다.
4. `12R SupplementStudentModal action ownership gate` — 12Q까지 표시·순수 계산·shell 분리는 완료했다. 다음은 local draft state와 makeup task/lesson 저장, `notification_jobs`/Solapi 예약·취소 handler 소유권을 App 밖으로 옮기는 고위험 단계다. 코드 이동 후 승인된 고태영 데이터로 저장·일정 생성/변경·세 알림 예약/취소·완료 처리와 새로고침 유지·중복 없음·대상 교차 없음을 다시 확인하기 전 다음 단위로 진행하지 않는다.

## Deferred Work Queue - Always Show First

새 세션은 아래 미룬 작업 큐를 최우선 최상단 작업지침으로 취급한다. 사용자가 별도 작업을 바로 요청하더라도, 코드 수정 전에 `AGENTS.md`, `docs/current-worklog.md`, `docs/next-session/README.md`, `git status --short`, `git log -1 --oneline`을 확인한 뒤 현재 미룬 작업과 새 요청의 우선순위를 먼저 짧게 정리해서 사용자에게 보여준다.

- 이 섹션과 `docs/current-worklog.md`의 `미룬 작업 큐`가 다음 작업의 source of truth다. 다음 세션용 붙여넣기 프롬프트에 일부가 빠져 있어도, 이 큐에 있으면 누락된 것이 아니라 계속 살아 있는 작업이다.
- 매 세션 종료 전, 새로 미룬 작업/보류 작업/다음 우선순위가 생기면 이 섹션, `docs/current-worklog.md`, 필요 시 `docs/next-session/README.md`를 함께 갱신한다.
- 사용자가 "다음 세션에 넘길 프롬프트"를 요청하면 이 큐를 프롬프트 최상단에 그대로 포함하고, 다음 세션이 시작하자마자 우선순위 목록을 사용자에게 보여주라고 명시한다.
- 미룬 작업은 한 번에 하나씩만 착수한다. 착수 전 저장 원천, 외부 side effect, 사람 gate, AI 자동검증 범위를 먼저 정리한다.

현재 미룬 작업 우선순위:

1. `운영 OS 저장 신뢰성 보강`
   - 상태: 2026-07-20 전수조사 완료. 상세 원천/화면/위험도는 `docs/save-persistence-audit-2026-07-20.md`가 기준이다.
   - 목표: 모든 운영 입력을 `local draft -> 명시적 저장/안전한 debounce -> API 완료 -> Supabase 재조회 -> 현재 영역 저장 상태` 구조로 통일한다.
   - UI 통일 목표: 저장 로직과 화면 표현을 따로 고치지 않는다. 모든 저장 화면은 `편집 중(미저장) -> 저장 중 -> 서버 반영 확인 중 -> 저장 완료` 또는 `저장 실패`의 같은 상태 모델을 사용하고, 버튼 문구·disabled 처리·현재 작업 영역의 상태 메시지·실패 시 draft 유지 방식도 공통화한다.
   - 성공 판정 원칙: localStorage나 낙관적 화면 변경은 저장 완료 근거가 아니다. API 성공과 필요한 Supabase 재조회/대조가 끝난 뒤에만 UI가 `저장 완료`를 표시하며, 새로고침 후 같은 값이 보이는 것까지 기능 단위 gate에 포함한다.
   - 저장·반영 계약: 저장 버튼마다 `직접 저장 원천`, `연결해서 갱신할 파생/소비 원천`, `외부 side effect`, `완료 재조회 범위`를 먼저 정한다. 사용자가 버튼 문구로 기대하는 연결 결과가 수업일지·달력·명단·포털 등에 반영되지 않았으면 직접 원천 저장이 성공해도 업무 완료로 표시하지 않는다.
   - 버튼 의미 원칙: 실제로 원천만 저장하는 버튼은 대상을 좁혀 `회차 계획만 저장`처럼 표시하고, 수업일지까지 반영하는 버튼은 `회차 계획 및 수업일지 반영`처럼 실제 범위를 말한다. 후속 반영이 필수라면 멀리 떨어진 별도 버튼으로 숨기지 않고 같은 작업 흐름에서 단계별 상태와 부분 실패를 표시한다.
   - 공통 UI 후보: 저장 가능한 모달/패널에는 같은 위치와 형태의 저장 버튼 및 `InlineSaveStatus` 계열 상태 표시를 사용한다. 자동저장이 필요한 화면도 동일한 상태 문구와 실패 복구를 제공하며, 단순 탭 전환/닫기 버튼과 저장 버튼을 시각적으로 구분한다.
   - 첫 착수 단위: `특강 회차 계획 저장 -> 안전한 미래 수업일지 명단 반영 통합`. `special_lecture_enrollments` 저장과 `lessons.studentIds/specialLectureStudentSchedules` 반영을 한 작업 흐름으로 묶되 원천별 성공/실패와 잠긴 회차를 분리 표시하고 두 원천을 재조회한 뒤에만 완료 처리한다. 2026-07-22에는 지난 회차 신청 원천 수정, 반복 수강시간 일괄 적용, 변경 사유 선택 입력, 기존 학생·시간을 보존하는 누락 학생 추가 확인 gate와 lessons 재조회 검증까지 보강했다. 신초봄 중3 원천의 1~5회차 13:00~15:00 운영 반영과 고3 오입력 enrollment 제외도 재조회 검증까지 완료했다. 그다음 `수업 생성/수정/복사/취소 저장 gate`를 진행한다.
   - 다음 순서: 학생 신규/Tally 정식등록/반 명단/퇴원 -> app_state key별 dirty 저장+version -> exam_prep/Tally 입력 경쟁 방지 -> 숙제/포털/자료함/보고서 -> 보충/알림 다중 원천 reconcile -> 오작동 버튼 정리.
   - 수업메모 저장 후속: 2026-07-23 메모·작성창 가져오기 2개·이전 확인 3개까지 Supabase 재조회 대조를 보강했다. 사람 gate 통과 후 `이전 메모 확인 범위/복구 -> X 닫기와 저장 의미 분리 -> 새로고침 draft 복구 -> 다중 탭 version 충돌 방지 -> compact 안내`를 별도 단위로 한 번에 하나씩 진행한다.
   - 중단 조건: 저장 보강이 `notification_jobs`/Solapi 실제 발송, 운영 데이터 삭제, 수업일지/출결 원천 변경까지 넓어지면 해당 단위를 중단하고 별도 사람 gate로 나눈다.
2. `모달 통일 작업`
   - 목표: 수업 생성/수업 상세/수업일지 관련 모달을 공통 구조로 정리하되, 정규수업, 숙제보충, 결석보충, 시험대비, 특강의 원천과 안내 정보를 잃지 않게 한다.
   - 현재 진행 상태: 숙제보충 등원보충 모달 1차 UI 정리, 결석보강 생성/상세 모달 1차 맥락 정리, 결석보강 알림톡 제목/본문 seed 정리, 실제 보강 수업 상세 화면과 생성 모달 문구 통일을 완료했다. 특강관리는 `Lesson Hub`로 이동했고 페이지 상위에 `특강 수업`과 `특강 안내문` 탭을 둔다. 신청자 원천/매칭/원본 카드 UI는 숨기고 확정 명단과 수업일지 반영 영역은 기본 접힘으로 정리했으며, 학생 수동 접수와 학생별 회차/진행 모달은 유지한다. 회차 모달은 로컬 draft와 Supabase 재조회 검증을 분리하고 내부에 저장 중/완료/실패 상태를 표시한다. 다음은 전체 모달 inventory와 나머지 공통 구조 통일 작업이다.
   - 수업일지 UI 후속: 하단 고정 저장 바 공통화가 끝난 뒤, 수업일지 상단의 `발송 전 점검/수정/예약 확인/기본 예약/30분 지연/알림톡 없음/Solapi 업데이트/발송결과` 기능은 유지하면서 CSS 위계와 접힘/그룹으로 정리한다. 저장 버튼과 Solapi side effect 버튼을 다시 한 묶음으로 합치지 않는다.
   - 필수 검토: 어떤 모달들이 있는지 먼저 inventory를 만들고, 각 모달이 읽고 쓰는 Supabase/app_state/local draft 원천, `notification_jobs`/Solapi side effect, 수업일지/숙제/출결 연결을 표로 정리한다.
   - 특히 보존할 정보: 숙제보충은 무엇을 언제 안 했고 무엇을 검사해야 하는지, 결석보충은 언제 결석했고 당시 확인해야 할 숙제/수업 내용이 무엇인지, 특강/시험대비는 정규수업과 지난 숙제/지난 수업 연결이 섞이지 않는지.
   - 중단 조건: 모달 통일 과정에서 알림톡 예약/발송, 출결 저장, 수업일지 기록 원천이 같이 바뀌면 즉시 중단하고 별도 gate로 쪼갠다.
3. `발송 알림톡 템플릿 설정 관리`
   - 목표: 실제 발송/예약되는 모든 알림톡 템플릿과 문구 원천을 `설정 > 알림톡`에서 확인하고 수정할 수 있게 한다.
   - 현재 진행 상태: 보충관리의 결석보강 학생 11시 리마인더, 숙제보충 학생 11시 리마인더, 보충 일정 확정 안내/변경 안내 문구는 `app_state.aiSettings.notificationTemplates` 원천으로 `설정 > 알림톡`에 노출했다. 나머지 발송 경로 전체 inventory와 설정 이관은 앞으로 해야 한다.
   - 추가 후속 작업: 수업 알림톡/데일리 리포트의 `⭐ 보충/확인 안내` 블록 문구를 다음에 수정한다. 착수 전 현재 문구 후보(`다음 수업 때 ... 함께 확인`, `오늘 수업 후 ... 보충`, 확정 보충/결석보강/재시험 일정 문구), 프론트 미리보기 경로, 서버 발송 직전 refresh 경로, 설정 화면 노출 여부를 먼저 표로 정리한다.
   - 필수 검토: 출결, 수업일지, 숙제보충, 결석보강, 재시험, 공지, 특강, 일정 변경, 학생 11시 리마인더 등 Solapi로 발송/예약되는 모든 경로를 inventory로 만들고, 각 템플릿의 현재 원천(하드코딩/app_state/env/Supabase), 변수, 대상(학생/학부모), 즉시발송/예약 여부를 표로 정리한다.
   - 원칙: 새 발송 경로를 추가하거나 기존 문구를 바꾸기 전에 해당 문구가 설정 화면에서 보이고 수정 가능한지 먼저 확인한다. 코드에 숨은 하드코딩 템플릿이 발견되면 기능 확장보다 설정 관리로 올리는 작업을 우선한다.
   - 중단 조건: 실제 발송되는 문구가 설정 화면의 미리보기/편집값과 다르거나, 설정 저장 없이 코드 상수만 바꿔야 하는 구조라면 즉시 중단하고 별도 gate로 쪼갠다.
4. `App.jsx 리팩터링 연속 작업`
   - 목표: App.jsx를 의미 단위로 계속 분리하되, 기능 변경과 리팩터링 범위를 섞지 않는다.
   - 순서: `원천/동작 보존 -> 파일 분리 -> 검증 명령 -> AI 검수 결과 + 사람이 확인할 것 gate -> 커밋/푸시`.
   - 우선순위: 위험이 낮은 helper/config/API/client/component부터 진행하고, `LessonJournalDetail`, 출결, Solapi 예약, 보충관리처럼 저장/발송 side effect가 큰 영역은 충분한 gate 이후 진행한다.
   - 기준 로드맵: 아래 `App.jsx Refactoring Roadmap - 18 Units`를 다음 세션의 리팩터링 후보 목록으로 사용한다. 이미 일부 분리된 항목도 남은 하위 컴포넌트/헬퍼가 있으면 같은 묶음 안에서 계속 쪼갠다.

## Parallel Refactoring + Maintenance Sessions - Required

사용자가 리팩터링과 유지보수를 별도 Codex 세션으로 나누는 동안에는 두 세션이 같은 Git worktree/index를 공유하지 않는다. 프롬프트만으로 같은 `E:\academy-os` 폴더의 동시 수정을 안전하게 만들 수 없으므로 아래 소유권을 우선한다.

- 유지보수 세션은 `E:\academy-os`의 `main` worktree와 `origin/main` 통합을 소유한다. 사용자 요청의 버그 수정/운영 개선만 진행하고 App.jsx 의미 단위 분리는 리팩터링 큐로 넘긴다.
- 리팩터링 세션은 별도 worktree(기본 예시 `E:\academy-os-refactor`)와 `codex/` 접두사의 전용 브랜치를 사용한다. 현재 이어받을 브랜치 예시는 `codex/refactor-supplement-11b`다. 이 세션은 유지보수 변경을 직접 만들거나 `main`에 merge/push하지 않고 전용 브랜치만 commit/push한다.
- 리팩터링 세션은 첫 작업과 각 새 의미 단위 시작 전에 `git fetch origin` 후 최신 `origin/main`을 기준으로 rebase/동기화한다. 예상하지 못한 변경, 충돌, 다른 세션의 staged 파일이 보이면 코드를 수정하지 말고 현재 상태와 충돌 파일을 사용자에게 보고한다.
- 두 세션 모두 다른 세션이 만든 파일을 stage/commit/revert하지 않는다. 특히 `src/app/App.jsx`, `src/app/App.css`, `package.json`, `scripts/scenario-tests-production.cjs`, `AGENTS.md`, `docs/current-worklog.md`, `docs/next-session/README.md`는 충돌 가능성이 높은 공유 파일로 취급한다.
- 리팩터링 브랜치의 통합은 유지보수 세션 또는 사용자가 명시적으로 맡은 한 세션만 수행한다. 통합 전 최신 `origin/main` 위에서 전체 production test와 build를 다시 실행하고, 리팩터링의 사람 gate가 보류라면 보류 상태를 그대로 유지한다.
- 기능 수정과 리팩터링이 같은 코드에 필요하면 유지보수 수정이 먼저 `main`에 들어간 뒤 리팩터링 브랜치가 이를 rebase한다. 리팩터링 브랜치가 유지보수 변경을 덮어쓰거나 과거 구현으로 되돌리면 안 된다.

## Workflow

0. 커밋/푸시는 최우선이다. AI가 할 수 있는 변경 검수, 커밋, 푸시는 사용자가 따로 말하지 않아도 작업 완료 즉시 자동으로 진행한다.
1. 작업 전 `docs/current-worklog.md`의 작업 큐를 확인하고 필요하면 갱신한다.
2. Supabase 저장은 최우선이다. 새 기능이나 화면 수정 전후로 데이터가 Supabase 테이블 또는 `app_state`에 저장되는지 먼저 확인한다.
3. 새로고침, 재로그인, 다른 기기 접속 후 사라질 수 있는 프론트-only/localStorage-only 운영 데이터가 있으면 기능 확장보다 저장 경로를 우선 보강한다.
4. DB 스키마 변경이 필요하면 `supabase/<file>.sql` 파일을 만들고, 적용이 필요한 SQL 파일명과 목적을 작업 결과에 명확히 남긴다. 운영 Supabase SQL edit 적용은 사용자가 직접 한다.
5. 한 번에 하나의 우선순위 작업만 구현한다.
6. 변경 범위는 요청된 화면과 데이터 흐름에 맞게 제한한다.
7. 구현 후 `npm run build`를 실행한다.
8. 운영 흐름에 영향을 주는 변경이면 `npm run test:production`도 실행한다.
9. 완료 후 작업 결과와 남은 이슈를 `docs/current-worklog.md`에 반영한다.
10. 검수 통과 후 GitHub `main`에 커밋/푸시해서 Vercel/Render 자동 배포가 이어지게 한다.
11. 사용자가 "다음 세션에 넘길 자료", "handoff", "인수인계 자료"를 요청하면 `docs/next-session/README.md`를 현재 최신 커밋, 검증 결과, 이어받을 작업 기준으로 새로 갱신한다. 다음 세션에는 이 폴더 하나만 넘기면 되도록 유지한다.
12. 작업 완료 답변에는 반드시 `사람 검토 절차`를 포함한다. 사용자가 실제 화면/데이터에서 무엇을 눌러보고 어떤 결과가 나오면 통과인지 순서대로 쓴다. 절차는 다음 단계로 넘어가기 위한 gate이며, 검토가 필요한 항목을 "나중에 확인"처럼 뭉뚱그리지 않는다.
13. 작업 완료 전 AI도 `AI 자기검토 질문`에 답할 수 있어야 한다. 단계별 버튼 안내만 맞고 전체 맥락/의도/저장 원천/다음 단계가 설명되지 않으면 완료로 보지 않는다.

## Refactoring + Maintenance Workflow - Required

App.jsx 리팩터링은 별도 일회성 프로젝트가 아니라 앞으로의 유지보수 개발 방식이다. 새 기능을 만들 때도 나중에 어떤 의미 단위로 분리될지, 어떤 저장 원천과 side effect를 건드리는지 같이 기록한다.

- 리팩터링은 한 번에 하나의 의미 단위만 진행한다. 기능을 고치려다가 리팩터링 범위가 커지면 현재 gate를 닫고 새 작업으로 분리한다.
- 리팩터링 단위는 `원천/동작 보존 -> 파일 분리 -> 검증 명령 -> AI 검수 결과 + 사람이 확인할 것 gate -> 커밋/푸시` 순서다.
- 일반 버그 수정/기능 구현은 기존 Workflow 0번처럼 검증 후 커밋/푸시가 기본이다. 단, 사용자가 리팩터링 연쇄 작업을 진행 중이라고 명시했거나, "사람 gate 후 커밋" 흐름을 요청한 경우에는 사람 검수 gate 통과 전 커밋/푸시하지 않는다.
- 각 리팩터링 종료 답변에는 고정된 gate가 아니라 실제 변경 범위에 맞는 `AI 검수 결과`와 `사람이 확인할 것`을 함께 작성한다. AI가 확인 가능한 정적/자동검증/side effect 검사는 먼저 수행하고, 사람 확인은 AI가 볼 수 없는 화면 어색함, 실제 운영 데이터, 외부 서비스 상태처럼 최소 항목으로 줄인다. 다음 세션으로 넘길 때도 "먼저 AI 검수 결과와 사람 확인 gate를 사용자에게 띄우고, 통과 확인 후 커밋/푸시" 순서를 명시한다.
- 새 기능 개발 중에도 다음을 함께 남긴다: 저장 원천, local draft 여부, API/DB 원천, notification/Solapi/Tally/Slack 같은 외부 side effect, 나중에 분리할 후보 파일명.
- 테스트 스크립트가 특정 문자열을 `App.jsx`에서만 찾고 있어 리팩터링을 막는 경우, 테스트의 의도는 유지하되 새 분리 파일까지 검사하도록 갱신한다. 테스트를 삭제하거나 기대 동작을 낮추지 않는다.
- App.jsx 리팩터링 우선순위는 위험이 낮은 helper/config/API/client/component 순서로 진행한다. `LessonJournalDetail`, 출결, Solapi 예약, 보충관리처럼 저장/발송 side effect가 큰 영역은 앞 단계의 gate가 충분히 쌓인 뒤 진행한다.

### App.jsx Refactoring Roadmap - 18 Units

이 목록은 다음 세션들이 공유하는 App.jsx 리팩터링 기준 순서다. 각 단계는 그대로 구현 지시가 아니라 `원천/동작 보존 -> 파일 분리 -> 검증 명령 -> AI 검수 결과 + 사람이 확인할 것 gate -> 커밋/푸시` 절차를 적용할 후보 묶음이다. 실제 착수 전에는 최신 `docs/current-worklog.md`와 `git status --short`를 보고 이미 끝난 하위 작업과 남은 하위 작업을 먼저 구분한다.

5. `specialLecture preview/public` - 특강 공개 안내문 미리보기, 공개 URL 렌더링, 복사 문구.
6. `specialLecture management` - Lesson Hub > 특강관리 편집 화면, 확정 명단, 특강 수업 생성 gate.
7. `school calendar helpers` - 학사일정/시험기간 날짜 계산, 표시 색상, 월간 요약 helper.

9. `test manager` - 시험지관리, 테스트 회차, 학생별 테스트 결과 입력/조회.
10. `student-parent portals` - 학생/학부모 포털, 공개 링크, 모바일 표시 컴포넌트.
11. `supplement job builders` - 숙제보충/결석보강 11시 예약, 일정 변경 알림, `notification_jobs` payload 생성.
12. `supplement center/modals` - 보충관리 화면, 숙제보충/결석보강 상세, 완료 처리.
13. `notification center` - 일반공지, 알림톡 관리, Solapi 발송/예약/결과 확인 UI.
14. `exam prep center` - 시험관리, 직전수업/시험대비 후보, 실제 수업 반영 gate.
15. `lesson hub/calendar` - 수업 달력, 수업 생성/수정/상세 모달, 자동 후보 표시.

## Project Essential Practices - Required

2026-07-04 시험분석 v2 재구성에서 효과가 컸던 문제 해결 방식을 프로젝트 필수 요소로 둔다. 새 세션은 아래 원칙을 단순 참고가 아니라 작업 기준으로 적용한다.

- 테스트 단계 기능이 2~3회 이상 보정에도 계속 꼬이거나, 저장 원천/AI 응답/화면 표시/임시 draft가 서로 다른 값을 말하면 필터를 더 덧대지 않는다. 먼저 `삭제 후 재생성`, `마이그레이션`, `원천 데이터 수정`, `기능 재구성` 중 어느 쪽이 맞는지 판단한다. 운영 데이터가 섞여 있으면 삭제 전 사용자 확인과 백업 기준이 먼저다.
- UI를 고치기 전에 데이터 층을 먼저 구분한다: 원본 row, 파생 표시값, AI 초안, local draft, 사용자 저장본, 최종 산출물. 어떤 층이 틀렸는지 확인하지 않은 채 화면에서만 맞아 보이게 하는 수정은 금지한다.
- PDF/AI/저장처럼 불확실성이 큰 작업은 작은 gate로 쪼갠다. 각 gate는 `입력`, `저장 위치`, `화면 확인 방법`, `새로고침 후 유지`, `실패 시 중단 조건`을 가져야 한다. 한 단계가 검증되기 전 다음 AI 단계로 넘어가지 않는다.
- AI가 문항 수, 행 개수, 저장 원본 같은 구조적 사실을 한 번에 추론해 결정하게 두지 않는다. AI 후보를 받은 뒤 사람이 확정하고, 이후 단계는 확정된 구조를 채우는 방식으로 진행한다.
- AI 초안은 사람이 볼 수 있는 편집 화면으로 노출한다. 사람 수정/저장 이후에는 그 값이 새 원본이다. AI 재생성, 자동 매핑, 렌더 재계산은 사람 편집본을 덮어쓰면 안 된다.
- 저장, 삭제, 업로드, AI 실행, 내보내기, 발송처럼 실패하거나 비용이 드는 작업은 현재 작업 패널 안에 진행/완료/실패 상태를 보여준다. 상태가 멀리 있거나, 차트만 바뀌어 저장된 것처럼 보이는 UX는 미완성으로 본다.
- 사용자가 스크린샷으로 "운영 화면에서는 아직 보인다"고 말하면 로컬 코드만 믿지 않는다. 운영 URL의 실제 번들, 최신 커밋, 배포/캐시 상태를 확인해 코드 문제와 배포 지연을 구분한다.
- 과금이 있는 AI 호출, 대량 삭제, 운영 데이터 변경은 자동 실행하지 않는다. 버튼, 상태, 사용자 검토 절차를 분리하고, 결과가 저장되는 위치를 명확히 남긴다.
- 작업 완료 시에는 성공한 것과 미룬 것을 분명히 구분한다. 테스트 단계라서 나중에 삭제 후 다시 돌릴 데이터, 품질 통과로 보지 않는 결과, 다음 단계에서 다시 봐야 할 항목을 숨기지 않는다.
- 사용자와 대화할 때 효과가 컸던 요청 형식을 존중한다: `검증가능한 절차를 알려주세요`, `코딩 전에 질문에 답해주세요`, `필터를 덧대지 마세요`, `AI 스스로도 답할 수 있는지 확인해주세요`, 스크린샷 기반의 짧은 증상 보고. 이런 요청이 나오면 구현보다 맥락 정리와 gate 설계를 먼저 한다.

## Core Development Direction - AI Draft To Human Source

Academy OS의 기본 개발 방향은 `AI 초안 -> 사람이 보기 쉬운 편집 화면 -> 사람 수정/검수 -> 검수본 원본화 -> 산출물/누적데이터 반영`이다. 이 방향은 시험분석, 수업일지 코멘트, 알림톡/문자, 블로그/인스타/PPT/Canva 산출물, 학생 누적 데이터 입력 화면에 공통 적용한다.

- AI는 먼저 데이터를 구조화하고 사용자가 한눈에 검수할 수 있는 화면을 만든다. AI 응답을 숨겨진 최종값으로 바로 쓰지 말고, 사람이 수정 가능한 draft로 노출한다.
- 새 AI/자동화 기능은 저장 레이어를 먼저 정한다: `ai_fields` 또는 동등한 AI 초안, 편집 중 `localDraft`, 저장된 `teacher_fields`/`human_fields`, 확정된 `final_fields`.
- 사용자가 수정하기 전에는 AI 초안이 편집 시작점이 될 수 있다. 사용자가 한 번이라도 수정하고 저장한 뒤에는 그 수정본이 새 원본이며, 이후 화면/새로고침/산출물/누적데이터는 `final_fields > teacher_fields/human_fields > ai_fields > 기본값` 순서로 읽는다.
- 편집 중 화면에서는 저장된 값보다 `localDraft`가 우선이다. 엔터, 중간 삽입, 커서 위치가 자연스럽게 유지되어야 하며, 렌더 중 파생값 재계산으로 input/textarea 값을 덮어쓰면 안 된다.
- AI 재생성, 자동 매핑, 템플릿 재적용은 빈 값, 명시적 재생성/초기화, 또는 사용자 동의가 있을 때만 허용한다. 사용자 편집본을 더 나은 AI 값으로 보이게 바꾸는 자동 덮어쓰기는 금지한다.
- AI가 만든 데이터를 시각화할 때는 사람이 수정할 수 있는 표, 선택값, 저장 버튼 또는 자동저장 상태를 함께 제공한다. 최종 미리보기는 검수본을 읽는 화면이지 AI가 숫자를 다시 만드는 화면이 아니다.
- 버튼이나 자동저장처럼 실패 가능성이 있는 동작은 현재 작업 영역 안에 `저장 중`, `저장 완료`, `저장 실패`, `AI 생성 중`, `AI 생성 완료` 같은 피드백을 둔다.
- 새 기능 구현 전 AI는 최소한 다음 질문에 답해야 한다: AI 초안은 어디에 저장되는가, local draft는 어디에 사는가, 어떤 API가 사람 수정본을 저장하는가, 저장 후 어떤 필드가 원본인가, 새로고침 후 사람 편집본이 AI 초안보다 우선하는가, 저장 상태는 어디에 보이는가.
- 이 구조를 지키기 위해 필터, normalize, fallback, 우선순위 보정을 여러 겹 덧대야 한다면 구현을 멈추고 원천 데이터 삭제/마이그레이션/재생성 중 무엇이 맞는지 사용자에게 확인한다.

## Autosave Risk Register - Required

자동저장은 Academy OS에서 운영 사고로 이어질 수 있는 고위험 영역이다. 새 세션은 자동저장 화면을 만들거나 수정하기 전에 아래 기준을 먼저 확인한다.

- 자동저장이 `localStorage`, Supabase/API 원본 저장, 알림톡 예약/발송, AI 초안 재생성 같은 서로 다른 층을 한 번에 움직이면 고위험으로 본다.
- 수업일지와 수업알림톡은 최우선 고위험 화면이다. 수업일지 입력 저장이 `lesson_student_records` 저장을 넘어 `notification_jobs` 갱신, Solapi 예약/취소, 발송 상태 patch까지 이어지면 자동저장 구조를 재검토한다.
- 전역 `app_state` 전체 snapshot 저장은 작은 변경이 여러 데이터 묶음을 덮어쓸 수 있으므로 고위험으로 본다. 가능한 경우 key별 dirty 저장, debounce, version/updatedAt 충돌 방지를 우선한다.
- 시험정보/시험 후 기록지, 학생 프로필, 보충 task, 학교 일정처럼 입력마다 API를 호출하는 화면은 중위험 이상으로 분류한다. 화면별 저장 버튼, 행 단위 저장, 또는 draft 후 일괄 저장을 검토한다.
- 자동저장 고위험 화면에는 후속 작업으로 붉은 경고 UI를 붙인다. 화면 상단이나 편집 영역 가까이에 `자동저장 위험` 배지를 표시하고, `왜 위험한가` 버튼을 눌렀을 때 저장 원천, 현재 부작용, 중단 조건, 권장 수정 방향을 보여준다.
- 붉은 경고 UI는 문제를 숨기기 위한 장식이 아니다. 해당 화면이 안전한 저장 구조로 바뀌면 경고를 제거하고 작업 기록에 제거 근거를 남긴다.
- 수업일지/수업알림톡의 권장 방향은 `수정 모드 -> local draft -> 저장 버튼 -> Supabase 저장 완료 확인 -> 별도 예약/발송 검수`다. 저장 버튼이 알림톡 예약/발송까지 자동으로 이어지면 안 된다.
- 일괄 저장 방식도 완전히 무위험은 아니다. 새로고침/탭 종료로 draft가 사라질 수 있고, 다중 탭 충돌, 일부 행 저장 실패, 오래된 데이터 덮어쓰기 위험이 있다. 따라서 변경 행만 저장, 행별 성공/실패 표시, 저장 전후 diff, 서버 updatedAt/version 확인, 발송 전 재검수 gate가 필요하다.

## Human Review Procedure - Required

작업이 끝나면 최종 답변에 아래 순서의 검토 절차를 포함한다. 코드 변경이 작아도, 문서만 바꿨어도 해당 범위에 맞게 축소해서 쓴다.

1. 버전 확인: 최신 커밋 해시, 배포 URL 또는 로컬 실행 위치, 검토할 화면/파일을 명시한다.
2. 데이터 준비: 사용할 테스트 데이터, 삭제/재생성해도 되는 데이터, 건드리면 안 되는 운영 데이터를 구분한다.
3. 기본 흐름 검토: 사용자가 화면에서 누를 버튼/입력할 값/기대 문구를 순서대로 적는다.
4. 저장 원천 검토: Supabase 테이블, `app_state`, Storage, localStorage 중 어디에 저장되어야 하는지와 새로고침 후 유지 여부를 확인하게 한다.
5. 실패/중단 조건: 저장 실패, 데이터 중복, 오래된 값 노출, 필터를 덧댄 보정이 보이면 다음 단계로 넘어가지 말고 멈추도록 쓴다.
6. 회귀 확인: 이번 변경과 가까운 기존 기능 1~3개를 골라 깨지지 않았는지 확인하게 한다.
7. 통과 기준: 모든 항목이 기대값과 일치해야 다음 단계로 진행한다고 명시한다. 애매하면 사용자 확인 후 진행한다.

## AI Self-Check - Required

작업을 완료하기 전에 AI는 아래 질문에 스스로 답할 수 있어야 한다. 기능/저장/API/UI를 바꾼 작업은 최종 답변에 짧은 `AI 자기검토`를 포함한다. 답할 수 없으면 더 읽거나 검증하고, 그래도 불명확하면 사용자에게 확인한다.

1. 전체 맥락: 이 작업은 사용자가 만들려는 큰 흐름의 어느 부분인가?
2. 사용자 의도: 사용자가 실제로 해결하려는 불편, 위험, 운영 목표는 무엇인가?
3. 변경 이유: 왜 지금 이 단계가 필요하며, 다음 단계로 무엇을 열어 주는가?
4. 저장 원천: 어떤 데이터가 원본이고, Supabase/app_state/Storage/local draft 중 어디에 저장되는가?
5. 사용자 편집본 보호: AI/템플릿/자동 매핑이 사용자 입력을 덮어쓰지 않는가?
6. 검토 가능성: 사용자가 화면에서 어떤 순서로 검토하면 성공/실패를 판단할 수 있는가?
7. 중단 조건: 어떤 현상이 보이면 다음 단계로 가지 말고 멈춰야 하는가?

## Exam Analysis Tab Review - Required When Touched

시험분석 탭, 시험분석 API, PDF 처리, Supabase v2 테이블, 분석 저장/삭제를 건드린 작업은 최종 답변의 `사람 검토 절차`에 아래 항목을 포함한다.

1. 시험분석 탭 진입: `학교 -> 학년 -> 고사 -> 분석` 카드가 보이고, 선택 카드/건수/상태 배지가 예상대로 바뀌는지 확인한다.
2. 테스트 데이터 기준: 시험분석 v2는 테스트 단계이므로, 검토에 쓸 분석/PDF가 삭제 후 재생성 가능한 것인지 먼저 밝힌다. 손상된 테스트 데이터는 보정하지 말고 삭제 후 재업로드를 우선한다.
3. PDF 업로드: 테스트 PDF를 업로드하고 `시험분석 PDF · 업로드 완료`가 보이는지, PDF 카드의 파일 크기/열기 링크가 맞는지 확인한다.
4. 텍스트 후보 추출: `텍스트 추출`을 눌러 쪽수, 추출 크기, 문항번호 후보, 누락 후보, 텍스트 레이어 잡음 경고가 보이는지 확인한다. 원문 프리뷰나 필터로 덮은 결과를 검토 기준으로 삼지 않는다.
5. AI 원본 검증: 사용자가 비용을 허용한 경우에만 `AI 검증(과금)`을 누른다. Claude/OpenAI 제공자, 쪽수, 과목, 문항 수 후보, 누락 번호가 표시되는지 확인하고, 문항별 크롭/유형분류가 자동 실행되지 않는지 확인한다.
6. 저장 원천 확인: `exam_analysis_runs`, `exam_analysis_sources`, `exam-analysis-pipeline-sources` Storage에 저장됐는지 확인하고, 새로고침 후 같은 분석/PDF/검증 상태가 유지되는지 확인한다.
7. 삭제 확인: 검토용 분석은 `삭제`로 지운 뒤 UI, DB row, Storage 객체가 함께 정리되는지 확인한다. 삭제가 필요한 테스트 데이터가 남아 있으면 다음 단계로 넘어가지 않는다.
8. 중단 조건: 과목이 엉뚱하게 바뀜, 중복 분석/중복 PDF가 생김, 옛 `app_state.examAnalyses` 경로가 살아남음, 저장 상태가 안 보임, 필터를 여러 겹 덧댄 보정이 필요함, 문항 수 후보가 원본과 불일치함. 이 경우 구현을 멈추고 사용자에게 확인한다.

- 중심 화면은 Lesson Hub / 수업일지이다.
- 학원명 표시 원칙은 `으뜸수학 고태영T`이다.

- 학생 화면은 태블릿 우선, 그다음 모바일이다.
- 학부모 화면은 모바일 우선이다.
- 알림톡 미리보기 문구는 실제 발송 문구와 최대한 같은 구조로 보여준다.
- API 모델 선택은 각 기능 화면에 반복 노출하지 말고 설정 화면에서 관리한다.
- 새 시험분석은 기존 삭제된 탭/`app_state.examAnalyses` 구조를 되살리지 않고 v2 파이프라인으로만 만든다. 순서는 `PDF 업로드 -> 문항 수 판독 -> 선생님 확인 -> 1~N 행 고정 -> AI 행 채움 -> 누락 검수 -> 재요청 -> 선생님 최종 확정`이다. AI가 문항 수나 행 개수를 결정하거나 선생님 수정본을 덮어쓰면 안 된다.
- 시험분석 v2는 테스트 단계이므로 기존 테스트 데이터가 손상되거나 원천 값이 틀렸으면 필터/정규화/우선순위 보정을 여러 겹 덧대어 숨기지 않는다. 필요하면 해당 테스트 데이터를 삭제하고 처음부터 다시 올리는 방향을 우선 검토한다. 원천 오류를 즉시 고칠 수 없거나 보정 로직이 2겹 이상 필요해 보이면 구현을 멈추고 사용자에게 확인한 뒤 진행한다.
- PDF/AI 분석 문제를 해결할 때 인코딩 잡음, 잘못된 과목, 중복 데이터 같은 증상을 문자열 필터 누적으로 처리하지 않는다. 먼저 원본 PDF, 저장 row, 파생 표시, AI 응답 중 어느 층이 원인인지 검증하고, 임시 필터는 명시적이고 제거 가능한 진단용으로만 둔다.
- 자동 매핑, AI, 템플릿으로 초안을 불러오는 기능은 "빈 값에 첫 초안을 채우는" 용도다. 사용자가 textarea/input에서 한 번이라도 수정한 뒤에는 그 사용자가 수정한 값이 새 초안이자 원본이다. 이후 렌더, 자동저장, 다른 필드 매핑, 정규화 로직이 사용자의 편집본을 다시 자동 초안으로 덮어쓰면 안 된다. 자동 갱신은 빈 값, 명시적 재생성/초기화 버튼, 또는 사용자가 동의한 경우로 제한한다.
- 자동 초안/자동 매핑 편집 영역은 커서 입력을 방해하면 안 된다. textarea 값은 매 렌더마다 파생값으로 재계산하지 말고, 사용자가 편집하는 동안에는 로컬 입력 상태 또는 동등한 안정 상태를 원본으로 삼아 줄바꿈, 중간 삽입, 커서 위치가 자연스럽게 유지되게 한다.
- 중요 지침: 자동 초안이 매 렌더마다 다시 계산되어 입력값을 덮는 구조는 금지한다. AI 초안, 템플릿 초안, DB/다른 필드 매핑값은 `초기 seed` 또는 명시적 `재생성` 이벤트에서만 편집 상태로 들어와야 한다. 이후 화면 렌더링은 안정적인 로컬 draft 상태, 저장된 사용자 편집본, 또는 명시적으로 선택된 source-of-truth를 읽어야 하며, JSX 렌더 본문에서 `value={deriveDraft(...)}`처럼 파생값을 직접 연결해 사용자의 타이핑/엔터/커서 위치를 깨뜨리면 안 된다.
- 중요 지침: 자동 초안 편집 UI를 만들 때는 `seed -> local draft -> save -> persisted teacher/user fields` 흐름을 먼저 설계한다. 저장 성공 뒤에는 서버가 돌려준 사용자 편집본으로 draft를 갱신하고, 새로고침 후에도 사용자 편집본이 AI/템플릿 초안보다 우선해야 한다. 이 원칙을 지키기 어렵거나 파생값 보정이 여러 겹 필요하면 구현을 멈추고 사용자에게 확인한다.
- 저장 버튼이 없거나 자동저장되는 편집 화면에는 사용자가 보고 있는 작업 영역 가까이에 `저장 중`, `저장 완료`, `저장 실패` 같은 상태를 분명하게 표시한다. 우상단처럼 시야에서 떨어진 위치에만 두지 말고, 가능하면 수업일지처럼 현재 편집 바/모달/패널 안에 표시한다. 어떤 데이터가 저장되는지 알 수 있게 `시험정보 · 저장 완료`, `수업일지 · 저장 중`처럼 대상 라벨을 함께 쓴다.
- 버튼이 저장, 삭제, 복사, 발송, 생성, 가져오기, 내보내기, AI 실행처럼 결과가 화면에 즉시 남지 않거나 실패할 수 있는 작업을 수행한다면 버튼 주변 또는 현재 작업 영역 안에 진행/성공/실패 상태를 반드시 표시한다. 탭 전환, 필터, 선택, 열기/닫기처럼 화면 상태가 바로 바뀌는 버튼은 그 시각적 변화 자체를 피드백으로 볼 수 있다.
- 학생 학습 화면은 단원별 유형을 노드형 지도처럼 보여주고, 학생이 각 유형의 이해/미완성/망각/실수 여부를 체크하며 복습할 수 있게 하는 방향을 우선 고려한다. 장기적으로 각 쎈 유형 노드는 풀이 영상, 주제별 교재, 헷갈리는 문항 묶음과 연결한다.
- 평가 운영 자동화는 누적테스트뿐 아니라 단원테스트, 주간테스트, 재시험까지 포괄한다. 출제 설계, 응시/OMR 제출 또는 자동채점, 채점 결과 데이터화, 오답지 생성, 재시험 배정, 재채점 이력을 하나의 흐름으로 연결하는 방향을 우선 고려한다.
- 테스트 결과 데이터는 학생별, 단원별, 쎈 유형별, 오답 주제별로 누적되어야 한다. 이후 수업일지, 보충관리, 학생 유형 노드맵과 연결할 수 있게 저장 경로를 먼저 설계한다.

## Exam Analysis Blog/Instagram/Canva Output Direction

- 시험분석 공개 산출물은 `공통 카드뉴스 템플릿 + 시험별 데이터 주입 + 선생님 검수 저장본` 흐름으로 만든다. 인스타는 카드뉴스만 올리고, 블로그는 같은 카드뉴스를 포함하되 더 상세한 해설/근거/학습 전략/상담 연결 문장을 붙인다.
- Canva/인스타 산출물 자동화 기준은 10장 카드뉴스다. 현재 Canva 링크의 실제 페이지 수가 10장이 아니면 자동화 기준을 8장 등으로 바꾸지 말고, Canva에 부족한 페이지를 추가해야 하는 별도 레이아웃 gate로 남긴다.
- 현재 Canva 디자인은 벤치마킹 블로그의 카드뉴스처럼 반복 사용 가능한 10장 템플릿인지 먼저 검토한다. 카드별 고정 역할, PNG 슬롯 크기, 텍스트 최대 길이, 대각선 리사이즈 가능 여부, 다음 시험에서 바꿀 데이터 영역과 고정 브랜드 영역을 구분한다.
- Canva 레이아웃은 `통렌더 카드 + 문제/손풀이 슬롯 카드`의 하이브리드로 본다. 수정이 거의 필요 없는 표지/구조/총평/다음대비/CTA 카드는 카드 전체를 통으로 렌더링하고, 실제 시험문제 원본이나 선생님 손풀이를 붙여야 하는 주요문항 카드에만 이미지 슬롯을 둔다.
- Canva 레이아웃 검토는 코딩과 별도 gate로 둔다. 1~10번 카드 중 어떤 카드는 통렌더이고 어떤 카드는 문제/손풀이 슬롯인지 구분해 모바일 가독성과 블로그 삽입 가독성을 확인한 뒤 웹앱 PNG 비율/크기를 조정한다.
- 블로그/인스타 AI 초안은 벤치마킹 포맷의 이모티콘 규칙을 따른다: 인사말 끝 `😊`, 주요문항 시작 `📌`, 핵심 포인트/자주 틀리는 지점 `✅`, CTA/신청 안내 `⬇️⬇️`, 위치 `📍`, 전화번호 `☎`.
- 블로그 초안은 18개 고정 섹션을 채우는 작업이 아니다. 사용자가 제시한 18개 흐름은 벤치마킹 글을 관찰한 예시 프리셋으로만 본다. 실제 구현은 `paragraph`, `highlight`, `cardImage`, `heading`, `questionExplanation`, `cta` 같은 블로그 블록(`blogBlocks`) 기반으로 설계한다.
- 의도 입력은 `이번 시험 난이도/점수 갈림/틀린 이유/다음 대비/학원 신뢰감` 5개 질문을 그대로 채우는 방식이 아니다. 이 5개는 최종 산출물이 답해야 하는 품질 체크이며, 선생님 입력은 카드 사이사이에 들어갈 블로그 흐름 블록과 주요문항 설명 중심으로 받는다.
- 주요문항 시험문제/손풀이 이미지는 자동 crop하지 않는다. 문항 수가 적으므로 사람이 직접 crop하고, JS/React 카드 렌더러는 선생님이 넣은 crop 이미지를 주요문항 카드 슬롯에 배치한다.
- 역할 구분: 사람은 원본 사실 확정, 주요문항 crop, 블로그 흐름 의도 입력, 최종 검수/공개 승인을 담당한다. AI는 문장 다듬기, 이모티콘/형광펜 위치 제안, 카드 문구 압축을 담당한다. JS/React 렌더러는 데이터 바인딩, 1080x1350 카드 미리보기, PNG/ZIP 생성을 담당한다. Canva는 제작 엔진이 아니라 최종 확인/예외 수정/업로드 보조 도구다.
- 카드뉴스/GPT Image 체크리스트는 매번 빈칸을 쓰게 하지 않는다. 웹앱 시험관리/총평 데이터가 `webAppExamSeed`, 시험지분석 v2의 문항 구조/단원 비중/난도 분포가 `examAnalysisSeed`, 선생님이 확인·수정·확정한 값이 `teacherConfirmChecklist`다.
- 객관식/서술형/단답형 수, 총점/배점, 단원별 실제 비중은 시험지분석 결과에서 자동 후보로 채운다. AI가 추측하거나 선생님에게 반복 입력을 요구하지 않는다. 단, 공개 산출물에는 선생님 검수/확정 후 사용한다.
- 주요문항은 시험지분석이 후보를 보여줄 수 있지만 최종 선정은 선생님이 한다. 블로그/인스타/카드뉴스/GPT Image 프롬프트는 `teacherConfirmChecklist`의 주요문항 최종 선택만 원본으로 사용하고, AI가 주요문항을 임의로 추가하거나 바꾸면 중단한다.
- 웹앱 산출물 입력 화면은 항목별 출처를 보여야 한다: `웹앱 자동 입력`, `시험지분석 후보`, `선생님 확정 필요`, `확정 완료`, `생성 금지`. 저장 후 새로고침에서는 선생님 확정값이 AI 후보보다 우선한다.
- AI가 할 수 있는 일은 이모티콘 위치, 줄나눔, 형광펜 태그, 문체 정리, 카드 문구 압축이다. AI가 해서는 안 되는 일은 문항 수/시험 범위/문항번호/유사문항/정답/풀이 사실을 추측하거나, 선생님 저장본을 덮어쓰거나, Canva 실제 레이아웃을 확인 없이 완료 처리하는 것이다.
- 벤치마킹 블로그의 전체 글 구조를 분석하되 그대로 복제하지 않는다. `제목 SEO -> 첫 문단 한줄 총평 -> 시험 구조 -> 변별 포인트 -> 주요문항 -> 다음 대비 -> 학원 관리 방식 -> CTA` 구조를 기본 틀로 삼고, `으뜸수학 고태영T` 톤에 맞게 더 블루/화이트 계열, 전문적이고 차분한 느낌으로 만든다.
- 블로그/인스타 카드뉴스는 같은 분석 원천을 쓰되 정보량을 다르게 한다. 인스타 카드에는 핵심 결론과 시각 자료를 압축하고, 마지막 카드에는 `더 자세한 시험 해설은 블로그에서 확인` 흐름으로 블로그 유입 CTA를 둔다. 블로그에는 카드뉴스 이미지와 함께 문항별 상세 해설, 학생이 틀릴 가능성, 다음 수업/보충 전략을 더 길게 붙인다.
- 학교별 변주는 템플릿을 깨는 디자인 변경보다 `학교명/학년/고사`, 대표 색 포인트 1개, 학교별 출제 습관, 주요문항 선택, CTA 문구 정도로 제한한다. 반복 제작 가능성이 깨질 만큼 카드 구조를 매번 바꾸지 않는다.
- 벤치마킹 블로그처럼 AI로 만드는 틀은 허용하되, AI 초안은 항상 사람이 볼 수 있는 편집 화면에 노출한다. 선생님이 수정/저장한 `teacherDraft`가 이후 블로그/인스타/Canva export의 원본이며, AI 재생성이나 템플릿 재적용이 이를 덮어쓰면 안 된다.

## Engineering Rules

- 비밀값은 절대 commit하지 않는다.
- `.env`, PDF, HWP, HWPX, ZIP, 대용량 자료는 Git에 올리지 않는다.
- Windows PowerShell 5.1은 UTF-8 무BOM 한글 파일을 기본 `Get-Content`로 읽으면 깨질 수 있다. 한글 문서를 읽을 때는 프로필의 UTF-8 기본값을 유지하거나 `-Encoding UTF8`을 명시한다.
- PowerShell 출력에서 한글이 깨질 수 있으므로 검수 테스트 이름은 가능하면 ASCII로 작성한다.
- 큰 파일 하나에 기능이 몰려 있어도, 이번 요청과 무관한 대규모 리팩터링은 피한다.
- 오류를 숨기는 패치 금지: 같은 문제를 막기 위해 필터, fallback, normalize, 우선순위 분기, 표시 보정이 여러 겹 쌓이기 시작하면 중단하고 원천 데이터/저장 경계/마이그레이션/삭제 중 어디서 해결할지 재검토한다. 사용자가 명시한 테스트 단계 데이터는 보존보다 원인 파악과 깨끗한 재시작을 우선한다.
- 사용자가 "검수"를 요청하면 버그/권한/저장 실패/데이터 원천 혼동/테스트 부족을 먼저 본다.
