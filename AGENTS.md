# Academy OS Agent Guide

이 파일은 Codex 새 세션이 이 저장소에서 작업할 때 가장 먼저 읽는 프로젝트 지침입니다.

## Read First

1. `docs/current-worklog.md`
2. `docs/home-codex-setup.md`
3. 관련 화면 코드와 데이터 파일

## Workflow

0. 커밋/푸시는 최우선이다. AI가 할 수 있는 변경 검수, 커밋, 푸시는 사용자가 따로 말하지 않아도 작업 완료 즉시 자동으로 진행한다.
1. 작업 전 `docs/current-worklog.md`의 작업 큐를 확인하고 필요하면 갱신한다.
2. Supabase 저장은 최우선이다. 새 기능이나 화면 수정 전후로 데이터가 Supabase 테이블 또는 `app_state`에 저장되는지 먼저 확인한다.
3. 새로고침, 재로그인, 다른 기기 접속 후 사라질 수 있는 프론트-only/localStorage-only 운영 데이터가 있으면 기능 확장보다 저장 경로를 우선 보강한다.
4. DB 스키마 변경이 필요하면 `supabase/<file>.sql` 파일을 만들고, 적용이 필요한 SQL 파일명과 목적을 작업 결과에 명확히 남긴다. 운영 Supabase SQL edit 적용은 사용자가 직접 한다.
5. 한 번에 하나의 우선순위 작업만 구현한다.
6. 변경 범위는 요청된 화면과 데이터 흐름에 맞게 제한한다.
7. 구현 후 `npm run build`를 실행한다.
8. 운영 흐름에 영향을 주는 변경이면 `npm run test:production`도 실행한다.
9. 완료 후 작업 결과와 남은 이슈를 `docs/current-worklog.md`에 반영한다.
10. 검수 통과 후 GitHub `main`에 커밋/푸시해서 Vercel/Render 자동 배포가 이어지게 한다.
11. 사용자가 "다음 세션에 넘길 자료", "handoff", "인수인계 자료"를 요청하면 `docs/next-session/README.md`를 현재 최신 커밋, 검증 결과, 이어받을 작업 기준으로 새로 갱신한다. 다음 세션에는 이 폴더 하나만 넘기면 되도록 유지한다.
12. 작업 완료 답변에는 반드시 `사람 검토 절차`를 포함한다. 사용자가 실제 화면/데이터에서 무엇을 눌러보고 어떤 결과가 나오면 통과인지 순서대로 쓴다. 절차는 다음 단계로 넘어가기 위한 gate이며, 검토가 필요한 항목을 "나중에 확인"처럼 뭉뚱그리지 않는다.

## Human Review Procedure - Required

작업이 끝나면 최종 답변에 아래 순서의 검토 절차를 포함한다. 코드 변경이 작아도, 문서만 바꿨어도 해당 범위에 맞게 축소해서 쓴다.

1. 버전 확인: 최신 커밋 해시, 배포 URL 또는 로컬 실행 위치, 검토할 화면/파일을 명시한다.
2. 데이터 준비: 사용할 테스트 데이터, 삭제/재생성해도 되는 데이터, 건드리면 안 되는 운영 데이터를 구분한다.
3. 기본 흐름 검토: 사용자가 화면에서 누를 버튼/입력할 값/기대 문구를 순서대로 적는다.
4. 저장 원천 검토: Supabase 테이블, `app_state`, Storage, localStorage 중 어디에 저장되어야 하는지와 새로고침 후 유지 여부를 확인하게 한다.
5. 실패/중단 조건: 저장 실패, 데이터 중복, 오래된 값 노출, 필터를 덧댄 보정이 보이면 다음 단계로 넘어가지 말고 멈추도록 쓴다.
6. 회귀 확인: 이번 변경과 가까운 기존 기능 1~3개를 골라 깨지지 않았는지 확인하게 한다.
7. 통과 기준: 모든 항목이 기대값과 일치해야 다음 단계로 진행한다고 명시한다. 애매하면 사용자 확인 후 진행한다.

## Exam Analysis Tab Review - Required When Touched

시험분석 탭, 시험분석 API, PDF 처리, Supabase v2 테이블, 분석 저장/삭제를 건드린 작업은 최종 답변의 `사람 검토 절차`에 아래 항목을 포함한다.

1. 시험분석 탭 진입: `학교 -> 학년 -> 고사 -> 분석` 카드가 보이고, 선택 카드/건수/상태 배지가 예상대로 바뀌는지 확인한다.
2. 테스트 데이터 기준: 시험분석 v2는 테스트 단계이므로, 검토에 쓸 분석/PDF가 삭제 후 재생성 가능한 것인지 먼저 밝힌다. 손상된 테스트 데이터는 보정하지 말고 삭제 후 재업로드를 우선한다.
3. PDF 업로드: 테스트 PDF를 업로드하고 `시험분석 PDF · 업로드 완료`가 보이는지, PDF 카드의 파일 크기/열기 링크가 맞는지 확인한다.
4. 텍스트 후보 추출: `텍스트 추출`을 눌러 쪽수, 추출 크기, 문항번호 후보, 누락 후보, 텍스트 레이어 잡음 경고가 보이는지 확인한다. 원문 프리뷰나 필터로 덮은 결과를 검토 기준으로 삼지 않는다.
5. AI 원본 검증: 사용자가 비용을 허용한 경우에만 `AI 검증(과금)`을 누른다. Claude/OpenAI 제공자, 쪽수, 과목, 문항 수 후보, 누락 번호가 표시되는지 확인하고, 문항별 크롭/유형분류가 자동 실행되지 않는지 확인한다.
6. 저장 원천 확인: `exam_analysis_runs`, `exam_analysis_sources`, `exam-analysis-pipeline-sources` Storage에 저장됐는지 확인하고, 새로고침 후 같은 분석/PDF/검증 상태가 유지되는지 확인한다.
7. 삭제 확인: 검토용 분석은 `삭제`로 지운 뒤 UI, DB row, Storage 객체가 함께 정리되는지 확인한다. 삭제가 필요한 테스트 데이터가 남아 있으면 다음 단계로 넘어가지 않는다.
8. 중단 조건: 과목이 엉뚱하게 바뀜, 중복 분석/중복 PDF가 생김, 옛 `app_state.examAnalyses` 경로가 살아남음, 저장 상태가 안 보임, 필터를 여러 겹 덧댄 보정이 필요함, 문항 수 후보가 원본과 불일치함. 이 경우 구현을 멈추고 사용자에게 확인한다.

## Product Rules

- 중심 화면은 Lesson Hub / 수업일지이다.
- 학원명 표시 원칙은 `으뜸수학 고태영T`이다.
- `koh_you_math`는 내부 식별자, URL, 환경변수 이름에만 남길 수 있다.
- 학생 화면은 태블릿 우선, 그다음 모바일이다.
- 학부모 화면은 모바일 우선이다.
- 알림톡 미리보기 문구는 실제 발송 문구와 최대한 같은 구조로 보여준다.
- API 모델 선택은 각 기능 화면에 반복 노출하지 말고 설정 화면에서 관리한다.
- 새 시험분석은 기존 삭제된 탭/`app_state.examAnalyses` 구조를 되살리지 않고 v2 파이프라인으로만 만든다. 순서는 `PDF 업로드 -> 문항 수 판독 -> 선생님 확인 -> 1~N 행 고정 -> AI 행 채움 -> 누락 검수 -> 재요청 -> 선생님 최종 확정`이다. AI가 문항 수나 행 개수를 결정하거나 선생님 수정본을 덮어쓰면 안 된다.
- 시험분석 v2는 테스트 단계이므로 기존 테스트 데이터가 손상되거나 원천 값이 틀렸으면 필터/정규화/우선순위 보정을 여러 겹 덧대어 숨기지 않는다. 필요하면 해당 테스트 데이터를 삭제하고 처음부터 다시 올리는 방향을 우선 검토한다. 원천 오류를 즉시 고칠 수 없거나 보정 로직이 2겹 이상 필요해 보이면 구현을 멈추고 사용자에게 확인한 뒤 진행한다.
- PDF/AI 분석 문제를 해결할 때 인코딩 잡음, 잘못된 과목, 중복 데이터 같은 증상을 문자열 필터 누적으로 처리하지 않는다. 먼저 원본 PDF, 저장 row, 파생 표시, AI 응답 중 어느 층이 원인인지 검증하고, 임시 필터는 명시적이고 제거 가능한 진단용으로만 둔다.
- 자동 매핑, AI, 템플릿으로 초안을 불러오는 기능은 "빈 값에 첫 초안을 채우는" 용도다. 사용자가 textarea/input에서 한 번이라도 수정한 뒤에는 그 사용자가 수정한 값이 새 초안이자 원본이다. 이후 렌더, 자동저장, 다른 필드 매핑, 정규화 로직이 사용자의 편집본을 다시 자동 초안으로 덮어쓰면 안 된다. 자동 갱신은 빈 값, 명시적 재생성/초기화 버튼, 또는 사용자가 동의한 경우로 제한한다.
- 자동 초안/자동 매핑 편집 영역은 커서 입력을 방해하면 안 된다. textarea 값은 매 렌더마다 파생값으로 재계산하지 말고, 사용자가 편집하는 동안에는 로컬 입력 상태 또는 동등한 안정 상태를 원본으로 삼아 줄바꿈, 중간 삽입, 커서 위치가 자연스럽게 유지되게 한다.
- 중요 지침: 자동 초안이 매 렌더마다 다시 계산되어 입력값을 덮는 구조는 금지한다. AI 초안, 템플릿 초안, DB/다른 필드 매핑값은 `초기 seed` 또는 명시적 `재생성` 이벤트에서만 편집 상태로 들어와야 한다. 이후 화면 렌더링은 안정적인 로컬 draft 상태, 저장된 사용자 편집본, 또는 명시적으로 선택된 source-of-truth를 읽어야 하며, JSX 렌더 본문에서 `value={deriveDraft(...)}`처럼 파생값을 직접 연결해 사용자의 타이핑/엔터/커서 위치를 깨뜨리면 안 된다.
- 중요 지침: 자동 초안 편집 UI를 만들 때는 `seed -> local draft -> save -> persisted teacher/user fields` 흐름을 먼저 설계한다. 저장 성공 뒤에는 서버가 돌려준 사용자 편집본으로 draft를 갱신하고, 새로고침 후에도 사용자 편집본이 AI/템플릿 초안보다 우선해야 한다. 이 원칙을 지키기 어렵거나 파생값 보정이 여러 겹 필요하면 구현을 멈추고 사용자에게 확인한다.
- 저장 버튼이 없거나 자동저장되는 편집 화면에는 사용자가 보고 있는 작업 영역 가까이에 `저장 중`, `저장 완료`, `저장 실패` 같은 상태를 분명하게 표시한다. 우상단처럼 시야에서 떨어진 위치에만 두지 말고, 가능하면 수업일지처럼 현재 편집 바/모달/패널 안에 표시한다. 어떤 데이터가 저장되는지 알 수 있게 `시험정보 · 저장 완료`, `수업일지 · 저장 중`처럼 대상 라벨을 함께 쓴다.
- 버튼이 저장, 삭제, 복사, 발송, 생성, 가져오기, 내보내기, AI 실행처럼 결과가 화면에 즉시 남지 않거나 실패할 수 있는 작업을 수행한다면 버튼 주변 또는 현재 작업 영역 안에 진행/성공/실패 상태를 반드시 표시한다. 탭 전환, 필터, 선택, 열기/닫기처럼 화면 상태가 바로 바뀌는 버튼은 그 시각적 변화 자체를 피드백으로 볼 수 있다.
- 학생 학습 화면은 단원별 유형을 노드형 지도처럼 보여주고, 학생이 각 유형의 이해/미완성/망각/실수 여부를 체크하며 복습할 수 있게 하는 방향을 우선 고려한다. 장기적으로 각 쎈 유형 노드는 풀이 영상, 주제별 교재, 헷갈리는 문항 묶음과 연결한다.
- 평가 운영 자동화는 누적테스트뿐 아니라 단원테스트, 주간테스트, 재시험까지 포괄한다. 출제 설계, 응시/OMR 제출 또는 자동채점, 채점 결과 데이터화, 오답지 생성, 재시험 배정, 재채점 이력을 하나의 흐름으로 연결하는 방향을 우선 고려한다.
- 테스트 결과 데이터는 학생별, 단원별, 쎈 유형별, 오답 주제별로 누적되어야 한다. 이후 수업일지, 보충관리, 학생 유형 노드맵과 연결할 수 있게 저장 경로를 먼저 설계한다.

## Engineering Rules

- 비밀값은 절대 commit하지 않는다.
- `.env`, PDF, HWP, HWPX, ZIP, 대용량 자료는 Git에 올리지 않는다.
- Windows PowerShell 5.1은 UTF-8 무BOM 한글 파일을 기본 `Get-Content`로 읽으면 깨질 수 있다. 한글 문서를 읽을 때는 프로필의 UTF-8 기본값을 유지하거나 `-Encoding UTF8`을 명시한다.
- PowerShell 출력에서 한글이 깨질 수 있으므로 검수 테스트 이름은 가능하면 ASCII로 작성한다.
- 큰 파일 하나에 기능이 몰려 있어도, 이번 요청과 무관한 대규모 리팩터링은 피한다.
- 오류를 숨기는 패치 금지: 같은 문제를 막기 위해 필터, fallback, normalize, 우선순위 분기, 표시 보정이 여러 겹 쌓이기 시작하면 중단하고 원천 데이터/저장 경계/마이그레이션/삭제 중 어디서 해결할지 재검토한다. 사용자가 명시한 테스트 단계 데이터는 보존보다 원인 파악과 깨끗한 재시작을 우선한다.
- 사용자가 "검수"를 요청하면 버그/권한/저장 실패/데이터 원천 혼동/테스트 부족을 먼저 본다.
